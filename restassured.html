# Agent Instructions for RestAssured API Testing Framework This document provides essential instructions for an AI agent to understand, navigate, and contribute to this Java RestAssured project. ## 1. Project Overview This is an automated API testing framework for the [Your Service Name] API. It uses Java, RestAssured, and TestNG to validate API endpoints. The primary goal is to ensure the correctness of API responses, including status codes, headers, and body payloads. ## 2. Core Technologies - **Language**: Java 17 - **Build Tool**: Maven - **API Testing**: RestAssured - **Test Runner**: TestNG - **Serialization**: Jackson (for JSON processing) - **Code Reduction**: Lombok - **Logging**: SLF4J with Logback ## 3. Getting Started & Running Tests The build lifecycle is managed by `pom.xml`. ### 3.1. Prerequisites - JDK 17 must be installed and configured. - Maven must be installed and in the system's PATH. ### 3.2. Key Maven Commands All commands should be run from the project's root directory. **Compile the project:** ```bash mvn clean compile ``` **Run all tests:** ```bash mvn clean test ``` **Run a specific test class:** ```bash mvn test -Dtest=com.yourcompany.tests.UserTests ``` **Run a single test method:** ```bash mvn test -Dtest=com.yourcompany.tests.UserTests#shouldCreateUserSuccessfully ``` **Run tests for a specific environment:** ```bash mvn clean test -Pstaging ``` ## 4. Project Structure The project follows standard Maven conventions. Key directories for your tasks are within `src/test/`. ``` src/test/ ├── java/ │ └── com/yourcompany/ │ ├── clients/ # API client classes (encapsulate endpoint logic) │ ├── config/ # Configuration loader and holder │ ├── models/ # POJOs for request/response bodies │ ├── tests/ # TestNG test classes │ └── utils/ # Shared utility functions (e.g., data generators) └── resources/ ├── config-dev.properties # Default environment configuration ├── config-staging.properties # Staging environment configuration └── logback-test.xml # Logging configuration pom.xml # Project dependencies and build configuration ``` ## 5. Core Abstractions & Design Patterns To maintain clean and reusable code, we use several key abstractions. Always use these patterns. ### 5.1. Configuration Management - Environment-specific details (e.g., `base.uri`, `api.key`) are stored in `.properties` files under `src/test/resources/`. - The `com.yourcompany.config.ConfigLoader` class reads the appropriate properties file at runtime based on the active Maven profile. - **NEVER** hardcode URLs, tokens, or other configuration in test classes. ### 5.2. Reusable Specifications (SpecBuilder) We use a builder pattern to create reusable `RequestSpecification` and `ResponseSpecification` objects. The `com.yourcompany.utils.SpecBuilder` class is the central place for this. **Example Usage:** ```java // In a test or client class RequestSpecification requestSpec = SpecBuilder.getRequestSpec(); ResponseSpecification responseSpec = SpecBuilder.getResponseSpec(); given().spec(requestSpec).body(userPayload) .when().post("/users") .then().spec(responseSpec).statusCode(201); ``` ### 5.3. API Clients For each API resource (e.g., Users, Products), there is a corresponding client class in the `com.yourcompany.clients` package (e.g., `UserClient.java`). This class encapsulates the RestAssured given/when/then logic for all operations on that resource. **Example UserClient.java method:** ```java public class UserClient { public static Response createUser(User userPayload) { return given().spec(SpecBuilder.getRequestSpec()) .body(userPayload) .when().post("/api/v1/users"); } } ``` ### 5.4. Data Models (POJOs) Request and response bodies are mapped to Plain Old Java Objects (POJOs) located in `com.yourcompany.models`. We use Lombok annotations (`@Data`, `@Builder`, `@NoArgsConstructor`, `@AllArgsConstructor`) to eliminate boilerplate code. Always create or use an existing POJO for JSON payloads. ## 6. Writing a New Test: Golden Path Follow these steps to add a new API test. **Scenario**: Add a test to verify fetching a user by ID. 1. **Check/Create POJO**: Ensure a `User.java` POJO exists in the models package to represent the user data in the response. 2. **Add Method to API Client**: Add a new static method to `UserClient.java`. ```java // In com.yourcompany.clients.UserClient.java public static Response getUser(String userId) { return given().spec(SpecBuilder.getRequestSpec()) .when().get("/api/v1/users/" + userId); } ``` 3. **Write the Test Method**: In the relevant test class (e.g., `UserTests.java`), add a new `@Test` method. ```java // In com.yourcompany.tests.UserTests.java @Test public void shouldGetUserByIdSuccessfully() { // Assume a user with ID "user-123" is created in a @BeforeMethod or exists in the test environment String userId = "user-123"; Response response = UserClient.getUser(userId); response.then() .spec(SpecBuilder.getResponseSpec()) // Verifies common assertions like Content-Type .statusCode(200); User user = response.as(User.class); assertEquals(user.getId(), userId); assertNotNull(user.getFirstName()); } ``` ## 7. Dependencies To add a new dependency, edit the `` section of the `pom.xml` file. After editing, let Maven resolve the new dependency. Run `mvn clean install` to confirm it works. ```xml com.new.library new-library-artifact 1.0.0 test ```
